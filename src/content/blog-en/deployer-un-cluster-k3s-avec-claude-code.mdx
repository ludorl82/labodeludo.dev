This week I took the single container host that ran my lab — including this site — and turned it into a real little Kubernetes cluster. This isn't an isolated project: for several months now, Claude Code has become my main tool for everything infrastructure-related here, both in interactive sessions, where I set the context and the decisions and it executes, diagnoses, and documents along the way, and for tasks that now run completely unsupervised. This k3s cluster project is a good chance to show concretely how that works on both sides.

## Why a cluster instead of a single server

A single container host is simple to reason about and easy to administer. It's also, by definition, a single point of failure: if that machine goes down, everything running on it goes down with it — the monitoring dashboard, this blog's deploy pipeline, all of it. With a k3s cluster spread across several physical machines, load can be redistributed if a node disappears, and nothing stops adding or removing nodes without restarting everything.

## The nodes: one foot in the cloud, one at home

The cluster has six nodes. The control plane runs on a cloud machine — that's **Captain**, the same instance I mentioned in a recent post, born specifically for this project. The other five nodes live at home: two Raspberry Pi 5s, and three virtual machines each running on a compute machine already in place in the lab. None of this has a port open directly on the Internet — the cluster talks over a private tunnel, like the rest of my home infrastructure.

## The startup gotchas

None of the five home machines agreed to start the k3s agent on the first try, and each for a different reason — a whole festival of small traps that Claude Code documented along the way, and that's worth repeating here.

The Raspberry Pi 5s run a recent Linux distribution where memory control groups (`cgroups`) aren't enabled by default at boot — a setting that has to be explicitly added to the boot configuration, followed by a reboot, before the k3s agent agrees to stay alive for more than a few seconds. The virtual machines built from minimal cloud images, meanwhile, simply didn't have `iptables` installed — the k3s install succeeded, but the service refused to start until the missing package was added. And one of the VMs thought it had barely 560MB of RAM because of a dynamic memory allocation setting misconfigured at creation time, nowhere near enough for a node — fixed once spotted.

None of it was hard on its own. The real trap was expecting a single install recipe to apply identically to all six machines — when in practice, every hardware type and every base image had its own little surprise in store.

## This site's deploy pipeline moves in

The site you're reading right now now also runs on this cluster. The continuous-integration runner, which builds and deploys the site on every change, now runs as an ordinary Kubernetes workload instead of a service installed permanently on a single machine — if the node hosting it disappears, the cluster can restart it elsewhere. The staging preview, the one I use to confirm an article renders correctly before publishing, moved from a shared file mount — which could, in rare cases, get overwritten by two overlapping deploys — to a dedicated volume per deploy, which eliminates that whole class of bug completely.

## What now runs on its own, at night

Last piece, and not the least important: every node in the cluster is now covered by an encrypted backup routine. Claude Code set it up and orchestrates it: weekly, at night, while everything sleeps, a script runs from the bastion and pushes an encrypted archive of every node to off-site storage.

But that's not all. The same week, a Claude Code instance started with no interface at all — "headless" mode, kicked off from a scheduled task on the bastion, with nothing for me to open — also handles system upgrades across the whole fleet: it connects to every machine in the lab, checks what needs updating, applies the patches, and documents what it did. It's the same agent, the same reasoning and command-execution loop as my interactive sessions, just triggered on a schedule instead of by me. I don't have to think about any of it anymore, and if a node disappears entirely, there's now a real safety net behind it instead of just hope.

## The economics: a subscription instead of tokens

A word on cost, because the question comes up often when I talk about this. For a long time I ran Claude Code on pay-per-use API billing — convenient to get started, but it can add up fast: a single heavy work day once cost me around $100 in tokens alone. I've since switched to the Claude Pro subscription, at $40 a month, which covers both my interactive sessions and headless work like the weekly upgrades described above. For usage as sustained as mine, the flat monthly subscription comes out well ahead of metered billing — and it completely removes the anxiety of checking the bill after a big work session.

## What we're taking away

-   A single container host is simple, but it's a single point of failure disguised as simplicity.
-   Every machine joining a cluster has its own small hidden startup ritual — don't assume a recipe that worked on the first one will apply unchanged to the next.
-   Moving the deploy pipeline onto the cluster eliminates bugs that only existed because everything shared the same machine.
-   Working through this kind of migration in session with an agent like Claude Code massively speeds up the execution and diagnostics side, but the architectural calls — how many nodes, where to place them, what to migrate first — stay mine.
-   The same agent can also run unsupervised, on a schedule, for recurring tasks like backups or system upgrades — without anyone having to remember them.
-   If usage is sustained, a flat subscription (Claude Pro, $40/month) beats pay-per-token billing by a wide margin — I've burned $100 in a single day on API alone.

The cluster is running, the site you're reading is on it, backups and upgrades happen while everyone sleeps, and the bill is predictable. Not bad for a week's work. — Ludo
