It started with a simple ask: does every server on my network actually have the IP address the documentation says it should? A routine check, the kind of thing you do now and then to keep things tidy. It ended, several hours later, with a cloud machine rebuilt from scratch. In between, enough pitfalls to fill an entire article — which works out nicely, since that's exactly what you're reading.

## A routine audit turns up a real bug

The k3s cluster running this site (see [the post on setting it up](/blog/deployer-un-cluster-k3s-avec-claude-code/)) has six nodes, each meant to sit in a specific address block by role — compute nodes in one range, storage in another, and so on. Comparing each machine's actual configured address against what the network documentation promised turned up a real problem: two of the home nodes had their DHCP reservations swapped in the firewall config. Node A was getting the address meant for node B and vice versa — probably a copy-paste mistake from when the second node was created, never noticed because both machines worked fine with "the wrong" address.

Nothing dramatic: two entries fixed in the firewall, and both nodes picked up their intended address at the next DHCP lease renewal. The kind of silent bug that never breaks anything until someone actually compares the two columns — documentation on one side, reality on the other.

## While we're at it, a little cosmetic cleanup

Real bug fixed, the temptation was too much: two other machines on the network had numbers that historically didn't quite make sense — the host running this site's deploy pipeline carried a number that, logically, should have belonged to a different machine, a leftover from an earlier rename. Nothing broken, just inelegant. I asked for the swap.

That's where it got interesting. The first sign, innocuous on its face: as soon as the new address landed in internal DNS, SSH attempts to that machine started warning about a changed host-key fingerprint — expected, since the name now pointed at an IP that, technically, still belonged to the other machine for a few more moments. A useful reminder that changing an IP address touches several systems that don't all catch up at the same speed.

The second pitfall was more subtle: one of the two machines flatly refused to take its new address until the previous occupant had fully vacated it — a standard duplicate-address-detection safeguard that, here, silently blocked the change with no clear error message. And even once the address was free, the DHCP server kept re-offering the same machine an old address it had already left behind — a stale lease sitting in the DHCP server's history, invisible unless you go digging directly in its lease database. The most pragmatic fix, in the end, was to pin the address statically rather than keep negotiating with the DHCP service.

One last detail, easy to overlook: k3s itself keeps each node's IP address hardcoded in its startup configuration. Changing a machine's address without updating that line makes the k3s agent crash on restart — it refuses to start with an address that no longer exists on any network interface. Easy to fix once you know where to look, easy to forget the first time.

## The node you can't just "rename"

Buoyed by two successful swaps, the next step seemed obvious: why not also align the control-plane node's address — the one living in the cloud — with the same tidy scheme? After all, it's just an IP address.

Except no. The cloud provider simply does not allow changing an already-created virtual machine's primary IP address — even after shutting it down first. It's a fundamental infrastructure constraint: that address is fixed at the moment the primary network interface is created, full stop. The only way to actually change it is to create a brand-new machine with the right address from the start, then migrate the role onto it.

That's the kind of wall you generally only discover by hitting it — and exactly the kind of discovery worth making in a lab rather than in an environment where that node hosts something more than a personal blog's deploy pipeline.

## Rebuilding a machine instead of editing it

Facing that wall, two options: drop the idea, or rebuild the machine entirely. The lab instinct won out over caution — let's see what that actually involves.

The procedure on paper is simple: cleanly stop the k3s service so as not to capture a half-written database, image the stopped machine, provision a new network interface with the right address, boot a new machine from that image, then move the public IP address over to the new machine. In practice, every step had its own surprise.

The first: for the new machine to keep exactly the same identity as the old one — same SSH key, no security warning on first login — you have to disable the mechanism that normally regenerates host keys on a "new" machine's first boot (cloud-init sees a fresh machine, since its internal identifier changes). That choice had an unexpected side effect: on this base image, that same mechanism is also responsible for adapting the network configuration to the new network card's physical address. Disabling it to preserve SSH keys left the new machine with a network configuration still pinned to the old network card — gone along with the old interface. Result: a machine that boots perfectly fine (confirmed via a console screenshot, since the boot logs themselves stayed oddly silent), but never gets any network at all.

Once the disk was fixed and reattached, the new machine booted with its network alive and well — plus a nice bonus: the encrypted private tunnel linking it back to the rest of the home network reconnected on its own, its private key having survived intact on the copied disk.

What remained was rewiring everything else that knew the old address by heart: k3s's own config on that node (which regenerated its security certificate on its own, no intervention needed — a nice touch), but also **each of the other five cluster nodes**, which each had the old address hardcoded as their connection point to the control plane rather than a domain name. Without that step, every node would have kept searching, in vain, for a machine that no longer exists at that address. The home firewall, for its part, needed three separate updates for the same address: its routing table, the encrypted tunnel's config (which only picked up the change after a full service restart — a plain config reload wasn't enough), and internal DNS shortcuts used by two or three other, unrelated services that happened to be hosted on that same node.

One last hiccup, minor but telling: during that whole unstable window, a service with no connection to the move got scheduled by mistake onto the control-plane node instead of a home node — and failed to mount its network storage, since that cloud node was never among the addresses allowed to reach that storage, regardless of which address it currently holds. Nothing to do with the address change itself — just a reminder that stirring up a cluster sometimes exposes old blind spots that had been quietly sleeping until then.

## The real safety net

Once everything checked out — all six nodes ready, the site reachable, the other services hosted on that node responding normally — the old machine stayed **stopped, not destroyed**, while everything was confirmed solid. Only once that confidence was earned did it actually get destroyed, along with the backup image created along the way. A safety net that costs almost nothing to keep around, and would have made all the difference had the rebuild gone sideways.

## Why a lab is the right place for this kind of adventure

Looking back at all this, one thing stands out: none of this chapter — swapping the other two nodes, fully rebuilding the control plane — was actually necessary. The cluster worked perfectly fine before, with addresses that didn't follow a perfectly tidy scheme but worked. In a professional context, a request like this — "let's renumber production servers to make things cleaner" — would almost certainly get turned down, and rightly so: the risk isn't justified by any functional gain.

That's exactly why a home lab has value. It's the place where you get to discover, with no real consequences, that a cloud provider won't let you change a machine's primary IP address — a constraint far better learned before you need to know it mid-crisis. It's the place where you can actually practice the rescue procedure — detach a disk, mount it elsewhere, fix a config file, reattach it — rather than reading about it for the first time in documentation while a real outage is running. And it's the place where the habit of keeping an old machine stopped instead of destroying it right away becomes muscle memory, not a box you realize afterward you should have checked.

The cluster now runs with a consistent numbering scheme end to end. But the real value of the exercise wasn't the tidiness of the result — it was every pitfall hit along the way, and getting to afford them.
